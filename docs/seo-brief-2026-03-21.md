## 2. Top 3 Keyword Opportunities

### 2a. "multi-agent AI memory" — Primary target

**Why:** This exact phrase represents the category Iranti is defining. It has low competition today because the space is new. Being the canonical result for this query while the category is forming is a strong position.

**Search intent:** Developer evaluating whether they need a dedicated memory layer for their multi-agent system. High purchase intent if they are already running multiple agents.

**Pages to target:**
- `/` (home) — primary
- `/product` — secondary

**Content gap:** The home page should contain the phrase "multi-agent AI memory" in a heading or prominent body text, not just metadata.

---

### 2b. "LLM memory infrastructure" — Positioning differentiator

**Why:** Iranti's core differentiator is that it is infrastructure, not a library or framework plugin. This phrase targets developers who have already moved past basic prompt engineering and are looking for a systematic solution. Low competition, high intent.

**Search intent:** Developer or architect planning a production system. They are past the "what is memory" stage and looking for a layer to commit to.

**Pages to target:**
- `/product` — primary (the vs. sections are ideal for this)
- `/` (home) — secondary

**Content gap:** The current `/product` heading uses "memory infrastructure, not another framework" — good. The phrase needs to appear earlier in the page and more literally in body copy.

**Why:** This phrase is searched by developers who have hit the problem: their agent forgets things across sessions. It maps directly to Iranti's primary guarantee. It is more concrete than "AI memory" and has weaker competition.

**Search intent:** Active problem — developer running an agent that loses state. Immediate evaluation intent.

**Pages to target:**
- `/get-started` — primary (conversion page)
- `/` (home) — secondary

**Content gap:** The install page should explicitly state "Iranti gives your agent persistent memory across sessions" in an h1 or h2 — not just in meta.

## 3. Comparison Page Opportunities

These are high-intent, relatively low-competition opportunities. Developers searching "Iranti vs X" or "X alternative" have already done primary research and are in final evaluation. These pages convert.

### 3a. `/vs/langchain-memory`
**Target queries:** "langchain memory alternative", "langchain memory limitations", "langchain persistent memory"

**Why:** LangChain is the dominant framework and its memory primitives are well-known to have limitations (framework-coupled, not portable, no cross-agent sharing). A detailed, honest comparison page wins this traffic because:
1. LangChain developers are actively looking for better solutions.
2. The query has purchase intent — they are already running LangChain.
3. Iranti has a validated integration (5/5 LangChain benchmark), lending credibility.

**Page structure:**
- What LangChain memory does well
- What it doesn't do (cross-agent sharing, persistence beyond a chain, conflict resolution)
- How Iranti fits alongside (not instead of) LangChain
- The 5/5 benchmark result
- Install CTA

---

### 3b. `/vs/memgpt`
**Target queries:** "memgpt alternative", "letta memory alternative", "memgpt multi-agent", "memgpt vs"

**Why:** MemGPT/Letta is the primary existing brand in the "AI memory" space. Developers researching agent memory will encounter it. A clear, honest comparison captures those evaluators at the moment of comparison.

**Key differentiation to make:**
- MemGPT gives one agent better memory. Iranti gives many agents shared memory.
- MemGPT is an agent architecture. Iranti is memory infrastructure you plug any agent into.
- Not a competitive attack — a positioning clarification. Both tools have valid use cases.

**Page structure:**
- MemGPT/Letta: what it is and what it does well
- The architectural difference (agent-internal vs. agent-external)
- When you need Iranti instead (or in addition)
- Framework-agnostic proof
- Install CTA

---

### 3c. `/vs/vector-db-memory` (or `/vs/pinecone-memory`)
**Target queries:** "vector database vs agent memory", "pinecone for agent memory", "weaviate agent memory alternative"

**Why:** Many developers reach for vector databases as their first memory solution. This is a teaching opportunity. The `/product` page already has a comparison table — a dedicated page with more depth and a clear narrative would rank for these queries.

**Key message:** Vector DBs answer "what's similar to X?" Iranti answers "what do we know about X?" These are different questions. You may need both.

## 6. Open Gaps — Not Addressed in This Session

- **OG image:** No `og:image` is set on any page. This means link previews on Twitter/X, Slack, and LinkedIn will show no image. The site needs a static OG image (1200×630) at `/public/og-image.png` and a reference added to `layout.tsx` metadata. This is the single highest-impact missing piece for social sharing.
- **Canonical URLs:** No `canonical` link tags are set. Next.js infers these from the URL, which is correct, but explicit canonical tags in metadata would be belt-and-suspenders for pages with potential duplicate URL variants.
- **Sitemap:** No `sitemap.xml` is present. Add a `src/app/sitemap.ts` Next.js dynamic sitemap. This is a one-file addition that materially helps crawl coverage.
- **robots.txt:** Not audited — verify one exists at `/public/robots.txt` with appropriate directives.
- **Structured data for Organization:** A `Organization` schema alongside the `SoftwareApplication` would help Google understand the entity behind the product. Low effort, moderate value.
